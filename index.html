Featured Project

Smart Transaction Capture Agent

An AI agent that auto-captures transactions from bank SMS/email alerts — built around the assumption that the parser will sometimes be wrong, and designed so that being wrong never silently corrupts the user's data.

01

The problem

Bank alert formats are inconsistent. A naive parser will misread promotional texts as purchases, double-count a transaction reported by both SMS and email, or fail on a garbled merchant string. Ship that naively, and a user's spending totals are silently wrong — with no way for them to know.

02

The gate

Every extraction passes through a Data Quality Gate before it can become real spending data. High-confidence, clean results are auto-accepted. Anything ambiguous — low confidence, a missing field, a possible duplicate — is held in a Review Queue instead of inserted automatically.

The core design principle: the AI is allowed to be uncertain, but it is never allowed to be silently wrong.
03

The proof

Evaluated against a labeled synthetic corpus covering every failure mode — clean transactions, duplicates, promotional noise, garbled merchants, and missing fields.

Without the gate, spend totals were off by $123.50 — entirely from one duplicate transaction silently double-counted. With the gate, the same dollar amount sat visibly in the Review Queue awaiting one tap, never corrupting anything. Same-size gap, completely different kind of wrong.

Why this approach

I started in software engineering before moving into product — which means I tend to ask "what happens when this is wrong" before I ask "what happens when this works." Most AI demos are built to show the happy path. I think the more useful question for a PM is the other one.

"Designing for the failure mode is the product decision — the model is just the implementation detail."
